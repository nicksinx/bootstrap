# Context

The OKF bundle should be readable by humans and useful to multiple agents without duplicating project knowledge into tool-specific files.

# Current Design

Project-1 uses this layout:

```text
Project-1/
  .okf/
  .claude/commands/       ← Claude Code project-local slash commands
  .cursor/rules/okf.mdc   ← Cursor project rule overlay (thin, generated)
  AGENTS.md               ← Codex adapter (thin, generated)
  CLAUDE.md               ← Claude Code adapter (thin, generated)
  docs/
  scripts/
  skills/                 ← Canonical OKF skill definitions (skills/*/SKILL.md)
  templates/
```

`.okf` is the primary context bundle. `AGENTS.md`, `CLAUDE.md`, `.cursor/rules/okf.mdc`, and `.claude/commands/` are thin adapters generated by `scripts/okf-sync-skills` from canonical skill definitions in `skills/*/SKILL.md`. `templates/` contains reusable starter files.

The `.okf/improvements/` directory is the continuous-improvement repository. It stores lessons learned, retrospectives, improvement actions, and reusable process changes discovered over the project lifetime.

# Shared Operating Model

Tool adapters are generated rather than hand-maintained. The sync flow is:

1. Edit the canonical skill definition in `skills/<name>/SKILL.md`.
2. Run `scripts/okf-sync-skills` to refresh all adapters.
3. Run `scripts/okf-validate` to confirm the bundle is valid.

Each adapter stays thin: it points agents to the canonical skill instead of duplicating the skill body. See `docs/shared-okf-skills.md` for the full operating model.

# Constraints

- OKF documents must stay human-readable.
- Tool adapters must stay thin.
- Scripts must avoid nonstandard dependencies.
- Secrets must not be stored in OKF.

# Interfaces

- Agents read `.okf/index.md` and `.okf/project.md`.
- Scripts operate on a repository-local `.okf` directory.
- `scripts/okf-sync-skills` generates adapters for Codex, Claude Code, Cursor, and future services.
- Codex installs skills by copying folders from `skills/` per `CODEX-SKILL-INSTALL.md`.
- Claude Code users can run `/project:okf-sync` to refresh adapters without leaving the editor.

# Dependencies

- Python 3 for local scripts.
- Markdown and YAML-compatible frontmatter conventions.

# Failure Modes

- Adapters drift from OKF when `scripts/okf-sync-skills` is not run after canonical skill changes.
- Context packs are mistaken for durable source material.
- Draft or unverified concepts are treated as accepted implementation input.
- Lessons learned remain in chat or handoffs and never become reusable project knowledge.

# Open Questions

- Whether portfolio-level OKF should maintain reusable skill versions across projects.
