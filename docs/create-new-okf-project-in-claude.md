# Create New OKF Project in Claude

Use this document when configuring **Claude Code as the third service** in a new OKF-enabled project. Cursor must complete step 1 and Codex must complete step 2 first, including the OKF scaffold, adapter configuration, any dispatch hooks, validation, and the Codex to Claude Code handoff.

This document is intended to be passed as context to Claude Code, Codex, Cursor, Xcode-connected Claude Agent, or a future service so every new project follows the same service setup order.

## Canonical service setup order

Configure services in this order for every new OKF project:

| Step | Service | Primary adapter / entry point | When it runs |
|------|---------|------------------------------|--------------|
| **1** | Cursor | `.cursor/rules/okf.mdc` | First — scaffold the project and OKF bundle |
| **2** | Codex | `AGENTS.md`, `.codex/hooks.json` | After Cursor handoff — install skills and Codex hooks |
| **3** | **Claude Code** | `CLAUDE.md`, `.claude/commands/` | After Codex handoff — wire Claude adapter and slash commands |
| **4** | Xcode | `.okf/agents/xcode-claude.md` or project equivalent | After Claude Code handoff — connect Xcode as a dispatch consumer |
| **5** | Perplexity Desktop Pro | `PERPLEXITY.md`, `.okf/agents/perplexity.md` | Last — cited deep research into `.okf/references/`; overflow failover |

Each step produces a handoff under `.okf/handoffs/` before the next service is configured. Do not skip ahead; later services assume earlier steps completed and validated.

- Step 1: `docs/create-new-okf-project-in-cursor.md`
- Step 2: `docs/create-new-okf-project-in-codex.md`

## What Claude Code does in step 3

Claude Code is responsible for making the project fully usable from Claude Code while preserving the OKF operating model established by Cursor and Codex.

Claude Code must:

1. Read the Codex handoff and confirm steps 1 and 2 are complete enough to continue.
2. Finalize the Claude Code adapter in `CLAUDE.md`.
3. Configure project-local slash commands under `.claude/commands/`.
4. Verify Claude Code's role in the dispatch pipeline when multi-agent delivery is enabled.
5. Update the Claude Code OKF agent concept and project log.
6. Validate the OKF bundle.
7. Write a Claude Code to Xcode handoff for step 4.

Claude Code does **not** fully configure Cursor global skills, Codex installed skill copies, or Xcode in step 3. It must not modify `.codex/hooks.json` or the canonical Codex adapter beyond what `scripts/okf-sync-skills` generates for the Codex target.

Claude Code has **no automatic hook mechanism** equivalent to Codex's `.codex/hooks.json`. Dispatch packet consumption by Claude Code is manual — Claude Code reads packets from `.okf/dispatch/ready/` when invoked, rather than firing automatically on session end.

## Prerequisites

Before Claude Code starts step 3, the project should already contain:

```text
.okf/
.cursor/rules/okf.mdc        ← active, thin, not modified in step 3
AGENTS.md                    ← refreshed by Codex step 2
CLAUDE.md                    ← stub ready for step 3 finalization
PERPLEXITY.md                ← stub generated by Codex; Claude Code verifies it exists
scripts/okf-validate
scripts/okf-sync-skills
scripts/okf-check-adapters
scripts/okf-context-pack
scripts/okf-handoff
skills/                      ← canonical OKF skill definitions
docs/okf-ways-of-working-brief.md
docs/create-new-okf-project-in-cursor.md
docs/create-new-okf-project-in-codex.md
```

If multi-agent delivery is planned, the project should also contain:

```text
scripts/okf-dispatch
.okf/dispatch/
  ready/
  running/
  done/
  failed/
docs/okf-dispatch-orchestration.md
.okf/workflows/multi-agent-delivery-pipeline.md
.codex/hooks.json            ← created in Codex step 2; do not modify in step 3
```

If these files are missing, Claude Code should record the gap in `.okf/log.md` and either repair the scaffold from the OKF bootstrap kit or stop with a handoff explaining what Codex needs to complete first.

## Step-by-step: configure Claude Code (service 3)

### 1. Open the project root in Claude Code

Open Claude Code (CLI or IDE extension) from the new project root. Claude Code reads `CLAUDE.md` automatically on session start.

For a quick initial read before any edits:

```bash
cat .okf/index.md
cat .okf/project.md
```

If the project is not yet a Git repository, initialize Git before serious work unless the project owner has chosen another path.

### 2. Read required context

Claude Code must read these files before editing:

1. `docs/okf-ways-of-working-brief.md`
2. `.okf/index.md`
3. `.okf/project.md`
4. Latest Cursor to Codex handoff under `.okf/handoffs/`
5. Latest Codex to Claude Code handoff under `.okf/handoffs/`
6. `docs/create-new-okf-project-in-cursor.md`
7. `docs/create-new-okf-project-in-codex.md`
8. `docs/shared-okf-skills.md` when present
9. `docs/okf-dispatch-orchestration.md` when dispatch is enabled
10. `docs/configure-perplexity-okf.md` when configuring a project for Perplexity (step 5)
11. `skills/create-new-okf-project/SKILL.md`
12. Relevant `skills/okf-*/SKILL.md` files for active tasks

Confirm the canonical service order in working notes:

```text
Cursor first, Codex second, Claude Code third, Xcode fourth, Perplexity fifth.
```

### 3. Confirm steps 1 and 2 complete

Before configuring Claude Code, verify:

- `.cursor/rules/okf.mdc` exists and remains a thin project-local overlay. Do not edit it — step 1 owns it.
- `.okf/index.md`, `.okf/project.md`, and `.okf/log.md` identify the new project.
- At least one requirement concept exists under `.okf/requirements/`.
- `AGENTS.md` was refreshed in step 2 and points to canonical skills.
- `PERPLEXITY.md` exists as a stub (generated by `scripts/okf-sync-skills --target perplexity`). If missing, generate it now before proceeding.
- A Codex to Claude Code handoff exists under `.okf/handoffs/`.
- `.codex/hooks.json` exists (when dispatch is enabled) and was created by Codex. Do not modify it in step 3.
- Cursor has not configured Claude Code or Xcode out of order.
- Codex has not fully configured Claude Code beyond the stub adapter and handoff instructions.

If Codex changed Claude Code-specific files beyond stubs, preserve useful work but re-align it to this document and the shared OKF model.

### 4. Refresh the Claude Code adapter

Run:

```bash
scripts/okf-sync-skills --target claude
```

This refreshes `CLAUDE.md` from canonical skills without duplicating full skill bodies into the adapter.

Check that `CLAUDE.md`:

- Points agents to `.okf/index.md` and `.okf/project.md`.
- Lists canonical `skills/*/SKILL.md` with short descriptions.
- Requires OKF log updates and handoffs after substantive work.
- Keeps secrets out of OKF.
- Does not duplicate large skill bodies.
- Is clearly marked as generated by `scripts/okf-sync-skills`.

Do not edit `.cursor/rules/okf.mdc` or `AGENTS.md` during step 3 unless the sync script must repair a broken generated adapter. Cursor remains service 1 and Codex remains service 2.

### 5. Configure Claude Code slash commands

Confirm `.claude/commands/okf-sync.md` exists. This was created by the scaffold script in step 1 (Cursor) but is officially **owned and verified by Claude Code in step 3**.

The command makes adapter sync available as `/project:okf-sync` inside Claude Code sessions. Check its content:

```bash
cat .claude/commands/okf-sync.md
```

The file should instruct Claude Code to:

1. Run `scripts/okf-sync-skills` from the project root.
2. Run `scripts/okf-validate` to confirm the bundle passes.
3. Append a short note to `.okf/log.md` describing what changed.

If the file is missing, create it:

```bash
mkdir -p .claude/commands
```

Then write `.claude/commands/okf-sync.md` with the content above. Keep the command thin — it invokes the script rather than duplicating sync logic inline.

Additional slash commands may be added for project-specific workflows (for example, context packs or handoffs), but each must remain a thin wrapper around a `scripts/` command or a canonical skill invocation.

Claude Code slash commands are **not** equivalent to Codex hooks. They are invoked manually during Claude Code sessions, not automatically on session end.

### 6. Configure Claude Code for dispatch

If the project uses the OKF dispatch queue, Claude Code participates as the **tester** role by default:

| Role | Default runner |
|------|----------------|
| `builder` | `codex` |
| `tester` | `claude` |
| `reviewer` | `cursor` |
| `integrator` | `codex` |

Claude Code reads typed work packets from `.okf/dispatch/ready/` for the `tester` role, executes the assigned work, and advances the packet state. See `docs/okf-dispatch-orchestration.md` for the full packet contract and runner mapping.

Claude Code must not invoke Codex, Cursor, or Xcode directly. It writes output packets and advances dispatch state by running `scripts/okf-dispatch` with the appropriate flags — it does not call other services.

**Important:** Claude Code has no automatic Stop hook. Dispatch advancement must be done explicitly during the session:

```bash
python3 scripts/okf-dispatch advance --from claude
```

If the project will rely on Claude Code for automated dispatch consumption, this constraint must be documented in the handoff for Xcode (step 4) so the overall delivery pipeline accounts for it.

Verify that `docs/okf-dispatch-orchestration.md` and the dispatch script agree on Claude Code's role. Do not change the role mapping during step 3 without updating both the orchestration doc and the relevant OKF workflow concept.

### 7. Update Claude Code OKF concepts

Review and update:

```text
.okf/agents/claude.md
```

The Claude Code agent concept should state:

- Claude Code is service 3 in the new-project setup order.
- Claude Code reads `.okf/index.md` and `CLAUDE.md` before substantive edits.
- Claude Code follows canonical `skills/*/SKILL.md` for OKF-related delivery.
- Claude Code participates as the `tester` role in the dispatch pipeline when enabled.
- Claude Code has no automatic hook — dispatch advancement is explicit during the session.
- Claude Code slash commands under `.claude/commands/` are thin wrappers; they do not replace canonical skills.
- Claude Code records validation evidence and handoffs.

If the concept was a draft placeholder, move it to `status: active` only after `CLAUDE.md`, `.claude/commands/`, and dispatch role verification are complete.

### 8. Validate

Run:

```bash
scripts/okf-validate
scripts/okf-check-adapters
```

If available, also preview the adapter in dry-run mode to confirm the sync script resolves canonical skills:

```bash
scripts/okf-sync-skills --target claude --dry-run
```

Record validation results in `.okf/log.md`. If validation fails, fix blocking issues or document them clearly in the handoff for Xcode.

### 9. Write the Claude Code to Xcode handoff

Create a handoff for step 4:

```bash
scripts/okf-handoff xcode-service-setup --summary "Claude Code step 3 complete; configure Xcode-connected Claude Agent as service 4."
```

Fill in the handoff with:

- **Current state** — Cursor step 1 complete, Codex step 2 complete, Claude Code adapter configured, validation result.
- **Completed work** — `CLAUDE.md`, `.claude/commands/`, dispatch role verification, `.okf/agents/claude.md` updated.
- **Decisions made** — service order confirmed, canonical skills remain in `skills/`, Claude Code dispatch is manual (no Stop hook), `/project:okf-sync` is the Claude Code sync entry point.
- **Files changed** — list Claude Code-related files.
- **Known issues** — any validation warnings, dispatch limitations, missing dispatch pieces.
- **Next recommended actions** — explicit Xcode step 4 checklist (see below).
- **Validation needed** — any Xcode-specific checks still pending.

## Handoff payload for step 4 (Xcode)

Include this checklist in the handoff so Xcode-connected Claude Agent receives consistent instructions across projects:

1. Read `.okf/index.md`, `.okf/project.md`, the Cursor handoff, the Codex handoff, and the Claude Code handoff.
2. Read `docs/create-new-okf-project-in-cursor.md`, `docs/create-new-okf-project-in-codex.md`, and `docs/create-new-okf-project-in-claude.md` for the canonical service order.
3. Confirm Cursor is step 1, Codex is step 2, Claude Code is step 3, and Xcode is step 4.
4. Create or verify `.okf/agents/xcode-claude.md` as the Xcode service adapter concept.
5. Connect Xcode-connected Claude Agent to the dispatch pipeline as the assigned role (typically `reviewer` or a project-specific role).
6. Do not modify `CLAUDE.md`, `AGENTS.md`, `.cursor/rules/okf.mdc`, or `.codex/hooks.json`.
7. Do not modify canonical skill definitions in `skills/*/SKILL.md`.
8. Run `scripts/okf-validate`.
9. Append to `.okf/log.md`.
10. Write a handoff confirming Xcode is configured as service 4 and noting that Perplexity Desktop Pro (step 5) is the next service to set up via `docs/create-new-okf-project-in-perplexity.md` and `docs/configure-perplexity-okf.md`.

## Claude Code step 3 completion checklist

Use this checklist before declaring step 3 done:

- [ ] Codex to Claude Code handoff was read.
- [ ] Canonical service order confirmed: Cursor, Codex, Claude Code, Xcode, Perplexity.
- [ ] Step 1 (Cursor) and step 2 (Codex) completion were verified before editing.
- [ ] `CLAUDE.md` was refreshed or verified via `scripts/okf-sync-skills --target claude`.
- [ ] `PERPLEXITY.md` was verified present (or generated via `scripts/okf-sync-skills --target perplexity`).
- [ ] `scripts/okf-check-adapters` was run and reported no drift.
- [ ] `.claude/commands/okf-sync.md` exists and invokes `scripts/okf-sync-skills`.
- [ ] Claude Code dispatch role (`tester`) is documented and verified in `docs/okf-dispatch-orchestration.md`.
- [ ] Claude Code's lack of an automatic Stop hook is recorded in the handoff for Xcode.
- [ ] `.okf/agents/claude.md` reflects Claude Code as service 3.
- [ ] `scripts/okf-validate` was run and the result recorded.
- [ ] `.okf/log.md` records Claude Code as the configured third service.
- [ ] A Claude Code to Xcode handoff exists under `.okf/handoffs/`.
- [ ] Xcode remains unconfigured except for thin stubs and handoff instructions.
- [ ] `.cursor/rules/okf.mdc`, `AGENTS.md`, and `.codex/hooks.json` were not modified beyond what sync scripts produce.

## Operating rules during step 3

- Treat `.okf` as the source of curated project context.
- Treat `skills/*/SKILL.md` as the canonical OKF skill definitions.
- Keep `CLAUDE.md` thin and generated where possible.
- Do not store secrets, API keys, tokens, private keys, signing credentials, or personal data in OKF.
- Do not modify Cursor global skills.
- Do not edit `.codex/hooks.json` or installed Codex skill copies.
- Do not fully configure Xcode during the Claude Code step.
- Do not let Claude Code slash commands become a second source of truth for skill logic — point to `skills/*/SKILL.md` instead.
- Advance dispatch manually when using the `tester` role; do not call other services directly.

## Quick reference commands (Claude Code step 3)

```bash
# Refresh Claude Code adapter from canonical skills
scripts/okf-sync-skills --target claude

# Preview without writing
scripts/okf-sync-skills --target claude --dry-run

# Validate OKF bundle
scripts/okf-validate

# Advance dispatch as tester (when dispatch is enabled)
python3 scripts/okf-dispatch advance --from claude

# Create handoff for Xcode
scripts/okf-handoff xcode-service-setup --summary "Claude Code step 3 complete."
```

Inside a Claude Code session, the `/project:okf-sync` slash command runs steps 1–3 of the sync workflow interactively.
