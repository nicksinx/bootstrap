## A. New product (bootstrap v2 launch)

Rules are included automatically when you launch with profile `default`:

```bash
cd /path/to/bootstrap
./scripts/launch_project.sh \
  --name my-product \
  --profile default \
  --target-dir ./out/my-product \
  --non-interactive

cd ./out/my-product
ls .cursor/rules/
# okf.mdc  okf-ecosystem-routing.mdc  okf-forge-operator.mdc  ...
```

Then refresh the skill index in `okf.mdc`:

```bash
scripts/okf-sync-skills
scripts/okf-validate
```

Reload the Cursor window (or restart Cursor) so rules are picked up.

## B. Existing product (manual install)

From a checkout of **bootstrap** (or copy from Project-1 `templates/tool-adapters/.cursor/rules/`):

```bash
BOOTSTRAP=/path/to/bootstrap
PRODUCT=/path/to/my-product

mkdir -p "$PRODUCT/.cursor/rules"

for f in \
  okf-ecosystem-routing.mdc \
  okf-legacy-aitask.mdc \
  okf-forge-operator.mdc \
  okf-dispatch.mdc \
  okf-forge-promotion.mdc
do
  cp "$BOOTSTRAP/templates/tool-adapters/.cursor/rules/$f" \
     "$PRODUCT/.cursor/rules/"
done

# okf.mdc: merge or replace — if you use okf-sync-skills, copy template then sync:
cp "$BOOTSTRAP/templates/tool-adapters/.cursor/rules/okf.mdc" \
   "$PRODUCT/.cursor/rules/okf.mdc"

cd "$PRODUCT"
scripts/okf-sync-skills    # refreshes skill list inside okf.mdc
scripts/okf-validate
```

**Do not** copy `bootstrap-toolkit.mdc` or `integration-harness.mdc` into product repos.

## E. Verify in Cursor

1. Open the project folder in Cursor (not a parent monorepo root unless rules live there).
2. **Settings → Rules** (or Command Palette → “Cursor Rules”): confirm project rules are listed.
3. Start a new Agent chat and ask: “Which layer should I use for lifecycle planning vs code delivery?” — the agent should cite Forge MCP vs `okf-dispatch` per `okf-ecosystem-routing.mdc`.
4. Open `scripts/conceptforge-mcp.sh` — scoped rules `okf-forge-operator` should appear in context.

## F. Maintenance

| Change | Action |
|--------|--------|
| Edit canonical rule wording | Update `templates/tool-adapters/.cursor/rules/*.mdc`, then copy to `templates/new-project/.cursor/rules/` and installed repos |
| Add OKF skills | `scripts/okf-sync-skills` (updates `okf.mdc` only) |
| Ship to all new products | Update `profiles/default.yaml` `files.include` (already lists rule paths) |
| Regression check | `make check` in bootstrap; `scripts/validate_launch.sh` in generated project |

**Important:** Static rules (`okf-ecosystem-routing`, forge, dispatch, promotion, legacy) are **not** regenerated by `okf-sync-skills`. Only `okf.mdc` is sync-managed.

---

## G. Legacy `legacy-task` profile

Products on profile `legacy-task` still receive v2 routing rules. `okf-legacy-aitask.mdc` activates only when editing `scripts/mcp/` or `scripts/workers/` and documents when legacy use is intentional.
