New Project Standup Guide

Step-by-step setup for creating a new project using Bootstrap outputs and pairing it with the live MCP server on GitHub.

Bootstrap Launcher GitHub MCP (PASS) Cursor-first workflow

Prerequisites

Local Tooling

  • git, gh, bash, python3
  • Optional: Node.js/npm for MCP local runtime checks

Access

  • GitHub account authenticated via gh auth status
  • Access to Bootstrap repo and MCP repo

Security Inputs

  • AITASK_HTTP_API_KEY
  • AITASK_HTTP_AUTH_MODE=all_requests
  • AITASK_MCP_API_KEY (for MCP tool auth)

Step-by-Step

1Clone Bootstrap and choose project id

Create a new working folder and pick a lowercase dashed project id.

git clone <your-bootstrap-repo-url> bootstrap
cd bootstrap
PROJECT_ID="acme-signals-api"
TARGET_DIR="$HOME/projects/$PROJECT_ID"
2Generate the project scaffold from Bootstrap

Run the launcher to generate project files, scripts, backlog, docs, and schemas.

./scripts/launch_project.sh \
  --name "$PROJECT_ID" \
  --profile default \
  --target-dir "$TARGET_DIR" \
  --non-interactive
Generated outputs include: scripts/, schemas/, backlog/, docs/, and project Makefile targets for validation and worker operations.
3Create and wire the GitHub repository

From the generated project directory, create a new GitHub repo and push initial code.

cd "$TARGET_DIR"
gh repo create "$PROJECT_ID" --private --source=. --remote=origin --push

If you need org ownership, use: gh repo create your-org/$PROJECT_ID --private --source=. --remote=origin --push.

4Configure secure MCP environment

Use hardened mode defaults for production-like usage.

export AITASK_HTTP_API_KEY="<strong-secret>"
export AITASK_HTTP_AUTH_MODE="all_requests"
export AITASK_MCP_API_KEY="<strong-secret>"
# Optional project allowlist:
export AITASK_MCP_ALLOWED_PROJECT_IDS="$PROJECT_ID"

Do not rely on insecure dev flags for shared or production environments.

5Attach MCP to the generated project

Generate MCP config and point to your live MCP server runtime mode (stdio or HTTP workflow).

./scripts/launch_project.sh \
  --name "$PROJECT_ID" \
  --profile default \
  --target-dir "$TARGET_DIR" \
  --with-mcp \
  --with-mcp-config \
  --non-interactive
Review generated .cursor/mcp.json and docs/mcp-integration.md in the project.
6Run validation and launch checks
make validate-launch
make test-launch
make release-gate

Expected: deterministic validation passes with no schema or queue invariant errors.

7Seed and dispatch initial backlog work
make backlog-init
make worker-next
make worker-status
make review-runs

This confirms Bootstrap outputs and worker contracts are live for the new project.

8Cursor operating pattern
  • Plan and approve in Cursor first.
  • Use MCP/project-scoped tools for execution and evidence retrieval.
  • Review generated artifacts before merge.
  • Keep secure MCP env vars in CI and deployment contexts.

Quick Verification Checklist

  • [ ] GitHub repo created and origin push succeeded
  • [ ] .cursor/mcp.json present and reviewed
  • [ ] make validate-launch passed
  • [ ] Worker dispatch/status/review flow succeeded
  • [ ] Secure MCP env vars configured (no insecure flags in shared env)

References

  • Bootstrap outputs: generated README.md, docs/agent-workflow.md, docs/mcp-integration.md
  • MCP release and readiness evidence: use your MCP repo’s docs/RELEASE_GATE.md and CI provenance artifact.
