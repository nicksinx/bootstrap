# Shared OKF Skills Operating Model

Project-1 uses a **project-local overlay** so Codex, Claude Code, Cursor, Perplexity, and future services can share the same OKF context, skill contracts, handoff format, and validation flow.

This model does **not** replace Cursor's existing global skills, rules, or workflows. Those remain valid. OKF adds a shared project memory layer on top.

## Canonical source

Canonical OKF skill definitions live in:

```text
skills/*/SKILL.md
```

Tool adapters stay thin. They point to these canonical skills instead of copying long skill bodies.

## Shared project context

Durable project knowledge belongs in `.okf`:

- requirements, features, architecture, decisions
- risks, tests, releases
- handoffs under `.okf/handoffs/`
- lessons learned under `.okf/improvements/`
- project log at `.okf/log.md`

Do not keep durable decisions, requirements, or handoff state only in chat.

## How each service participates

| Service | Adapter | Operator skill | Role |
|---------|---------|----------------|------|
| Cursor | `.cursor/rules/okf.mdc` | `skills/cursor-okf-operator/SKILL.md` | Reviewer; **Forge MCP host**; promotion integrator |
| Codex | `AGENTS.md` | `skills/codex-okf-operator/SKILL.md` | Builder / integrator; dispatch hooks |
| Claude Code | `CLAUDE.md` | `skills/claude-okf-operator/SKILL.md` | Tester; dispatch consume |
| Xcode | `.okf/agents/xcode-claude.md` | `skills/xcode-okf-operator/SKILL.md` | Platform dispatch consumer |
| Perplexity | `PERPLEXITY.md` | `skills/perplexity-okf-*/SKILL.md` | Research + overflow (**no Forge MCP**) |

Canonical cross-service skills (`okf-reader`, `okf-handoff-writer`, etc.) apply to all services. **Service operator skills** add dispatch role, layer boundaries, and v2 Forge hosting policy without duplicating Forge MCP in every IDE.

See `docs/okf-service-operator-skills.md`.

Configure Perplexity: `docs/configure-perplexity-okf.md`

## Perplexity (service 5 + overflow)

Perplexity Desktop Pro has two OKF modes:

| Mode | When | Output |
|------|------|--------|
| **Research (service 5)** | Planned external evidence | Draft Reference concepts in `.okf/references/` (`unverified`) |
| **Overflow (ad hoc)** | Primary runner blocked by limits/outage | Same role deliverable for Cursor/Codex to apply |

1. Paste custom instructions from `.okf/prompts/perplexity-custom-instructions.md` into Perplexity Desktop settings.
2. For research, use **MODE A** and `docs/create-new-okf-project-in-perplexity.md`.
3. For failover, use **MODE B** and `.okf/prompts/perplexity-overflow-failover.md`.
4. Cursor or Codex always ingests outputs into the repo and runs `scripts/okf-validate`.

Perplexity does not replace Cursor global skills or execute dispatch hooks directly.

Perplexity Project Skills are service-specific adapters, not the same as Codex-installed canonical skills:

- `skills/perplexity-okf-reader/SKILL.md`
- `skills/perplexity-okf-citation-steward/SKILL.md`
- `skills/perplexity-okf-handoff-writer/SKILL.md`
- `skills/perplexity-okf-concept-writer/SKILL.md`

Their frontmatter includes `applies_to`, `okf_mode`, and `canonical_skill` so Perplexity Desktop can use them as project-attached skills while still pointing back to the canonical OKF contracts.

## Cursor usage (overlay, not replacement)

When working in Cursor on this project:

1. Keep using Cursor's existing global skills when they fit the task.
2. Read `.okf/index.md` and `.okf/project.md` before substantive project changes.
3. For OKF-related delivery, open and follow the matching file in `skills/*/SKILL.md`.
4. Update `.okf` and append to `.okf/log.md` after substantive changes.
5. Create a handoff in `.okf/handoffs/` before pausing work another agent may continue.

The project rule at `.cursor/rules/okf.mdc` is regenerated by the sync script and should remain thin.

## Handoffs across services

Any agent can pause and another can continue if work follows the same contract:

1. Read `.okf/index.md`, `.okf/project.md`, and recent `.okf/handoffs/`.
2. Use `skills/okf-reader/SKILL.md` for triage when needed.
3. Implement using the relevant canonical skill.
4. Update OKF concepts, append to `.okf/log.md`, and write a handoff with `skills/okf-handoff-writer/SKILL.md` or `scripts/okf-handoff`.
5. Run `scripts/okf-validate` before commit or release when practical.

Context packs (`scripts/okf-context-pack` or `skills/okf-context-pack-builder/SKILL.md`) are generated task context, not durable source material.

## Propagating canonical skill changes

When a canonical skill under `skills/*/SKILL.md` changes:

1. Edit the canonical `SKILL.md` only once.
2. Run:

```bash
scripts/okf-sync-skills
```

3. Run validation:

```bash
scripts/okf-validate
```

4. Append a short note to `.okf/log.md`.
5. For Codex-only install mechanics, see `CODEX-SKILL-INSTALL.md`.

Useful sync options:

```bash
scripts/okf-sync-skills --dry-run
scripts/okf-sync-skills --list-skills
scripts/okf-sync-skills --target cursor
scripts/okf-sync-skills --target perplexity
scripts/okf-check-adapters
scripts/okf-check-adapters --strict
```

## Assumptions about Cursor project rules

- Cursor project rules live under `.cursor/rules/*.mdc`.
- `alwaysApply: true` on `.cursor/rules/okf.mdc` makes OKF context available without disabling other skills.
- Cursor global skills under the user's Cursor config are outside this repository and intentionally untouched.
